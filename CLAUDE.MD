## Product

**GameTrackr** is a social game-tracking app ‚Äî think Backloggd / Letterboxd, but for video
games. A user can:

- Log the games they own and track their status (playing, completed, backlog, platinum, abandoned, wishlist)
- Rate games, log hours, start/finish dates, and write short reviews
- Discover games (new releases, most anticipated, trending)
- Build a public profile with stats (total games, hours, platinum rate)
- Add friends, visit profiles, and message in real time
- Join a light community (topics, comments, votes), optionally tied to a specific game
- Catalog their physical setup (consoles, peripherals, collectibles)

The mobile-native clients and the web client are **thin clients** over one backend.

## External game data

Game metadata (covers, release dates, genres, hype, screenshots) comes from **IGDB** and/or
**RAWG**, cached **server-side** in the backend's own `games` table. Clients **never** call
IGDB/RAWG directly ‚Äî they only ever talk to the GameTrackr API. The backend owns refreshing
and caching that data.

## Authentication model

- **Token-based** auth via **Laravel Sanctum** (Bearer token in `Authorization` header).
- Mobile stores tokens in **secure storage** ‚Äî Keychain on iOS, EncryptedSharedPreferences / encrypted DataStore on Android. Never in plain UserDefaults / SharedPreferences.
- **Social login** (Google, Apple, Steam, Discord): OAuth is handled by the backend; the
  mobile client opens the provider flow (system browser / ASWebAuthenticationSession /
  Custom Tabs) and receives a GameTrackr token at the end.
- Whether refresh tokens are used (vs. long-lived Sanctum tokens) is **to be confirmed with backend** ‚Äî see the open questions section.

> ‚úÖ **VERIFIED AGAINST THE RUNNING BACKEND (2026-06-27).** The text above is the original
> *proposal*; the points below were confirmed with `curl` against the local backend and **take
> precedence** where they differ:
> - Auth is **JWT (tymon/jwt-auth), NOT Sanctum.** `JWT_TTL` 60 min, `JWT_REFRESH_TTL` 14 days.
> - Refresh = **single-token rotation**: on `401`, `POST /auth/refresh` with the *current* token in
>   the header (no body) ‚Üí returns a new token, blacklists the old one. **No separate refresh token.**
> - Verified endpoints differ from the proposed list below: `POST /auth/register` ¬∑
>   `POST /auth/login` ¬∑ `POST /auth/refresh` ¬∑ `POST /auth/validate` ¬∑ `POST /auth/logout` ¬∑
>   `GET /profile/me` (not `GET /me`). `password/forgot|reset` **do not exist yet**.
> - `register` body is **`{ name, email, password, password_confirmation }`** (Laravel `confirmed`
>   rule) ‚Äî the field is **`name`**, not `username`.
> - Token storage on Android is **DataStore (Preferences)**, not EncryptedSharedPreferences
>   (the Jetpack Security Crypto lib is deprecated). iOS still uses the Keychain.

## Realtime

Powered by **Laravel Reverb** (Pusher protocol). Mobile consumes it with a Pusher-compatible
client (`pusher-websocket-java` on Android, `pusher-websocket-swift` / `PusherSwift` on iOS),
or a raw WebSocket if the team prefers.

Channels (proposed):

- `private-user.{id}` ‚Äî notifications, friend requests
- `private-conversation.{id}` ‚Äî direct messages
- `private-user.{id}.activity` ‚Äî activity feed updates

## Proposed REST contract

Base URL: `https://api.gametrackr.app` (prod) / configurable per build for local/staging.
All authenticated requests send `Authorization: Bearer <token>`.
JSON request/response. Pagination via `?page=` + `?per_page=` returning Laravel-style
`{ data, meta, links }`.

### Auth
> ‚öÝÔ∏è Proposed shape below; see the ‚úÖ **verified** note under *Authentication model* for what the
> running backend actually exposes (JWT, `name`+`password_confirmation`, `auth/refresh`,
> `auth/validate`, `profile/me`). The lines below are kept as the original team proposal.

- `POST /auth/register` ‚Äî `{ username, email, password }` ‚Üí user + token *(verified: `{ name, email, password, password_confirmation }`)*
- `POST /auth/login` ‚Äî `{ email, password }` ‚Üí user + token
- `POST /auth/logout` ‚Äî invalidate current token
- `POST /auth/password/forgot` ‚Äî `{ email }` ‚Üí 204 (sends reset email) *(verified: does not exist yet)*
- `POST /auth/password/reset` ‚Äî `{ token, password }` ‚Üí 204 *(verified: does not exist yet)*
- `POST /auth/email/verify` ‚Äî `{ code }` ‚Üí 204 (if email OTP is used)
- `POST /auth/social/{provider}` ‚Äî provider in `google|apple|steam|discord` ‚Üí token
- `GET  /me` ‚Äî current user + profile *(verified: `GET /profile/me`; also `POST /auth/refresh` + `POST /auth/validate`)*

> **Sign-up form (shared by all clients).** The Create-account / sign-up screen collects
> **Name ‚Üí `username`**, **Email**, **Password**, **Confirm password**, plus a **required**
> Terms of Use / Privacy Policy acceptance. Client-side validation is identical on every
> client: name ‚â• 3 chars, email contains `@` and `.`, password ‚â• 6 chars, confirm must match,
> terms must be checked. Field errors surface only **after the first submit**. A
> password-strength meter (weak / medium / strong) is shown as the user types. Social sign-up
> currently exposes **Continue with Google** only ‚Äî Apple / Steam / Discord are deferred.

### Games (discovery)
- `GET /games/releases` ‚Äî recent releases
- `GET /games/upcoming` ‚Äî most anticipated (with hype/date)
- `GET /games/trending` ‚Äî trending (server-computed from platform activity)
- `GET /games/search?q=` ‚Äî search the cached catalog
- `GET /games/{slug}` ‚Äî game detail

### Library
- `GET    /me/library?status=&platform=&sort=` ‚Äî user's library, filterable
- `POST   /me/library` ‚Äî `{ game_id, status }` ‚Äî add a game
- `PATCH  /me/library/{id}` ‚Äî update status, rating, hours, dates, review
- `DELETE /me/library/{id}` ‚Äî remove
- `GET    /me/stats` ‚Äî aggregated stats (totals, platinum rate, by genre/platform/year)

### Profiles & friends
- `GET  /users/{username}` ‚Äî public profile + public library
- `GET  /users/{username}/stats`
- `POST /friends/requests` ‚Äî `{ user_id }`
- `POST /friends/requests/{id}/accept`
- `POST /friends/requests/{id}/decline`
- `GET  /friends` ‚Äî friend list
- `DELETE /friends/{id}`

### Messaging (realtime-backed)
- `GET  /conversations` ‚Äî list
- `GET  /conversations/{id}/messages?before=` ‚Äî paginated history
- `POST /conversations/{id}/messages` ‚Äî `{ body }`
- `POST /conversations` ‚Äî `{ user_id }` ‚Äî start/find a 1-1 conversation

### Community
- `GET  /topics?game_id=&category=` ‚Äî list
- `POST /topics` ‚Äî `{ title, body, game_id? , category }`
- `GET  /topics/{id}` ‚Äî topic + posts
- `POST /topics/{id}/posts` ‚Äî `{ body }`
- `POST /posts/{id}/vote` ‚Äî `{ value: 1 | -1 }`

### Collection (physical)
- `GET    /me/collection`
- `POST   /me/collection` ‚Äî multipart: `{ category, name, description?, image? }`
- `PATCH  /me/collection/{id}`
- `DELETE /me/collection/{id}`

### Notifications & feed
- `GET  /notifications` ‚Äî list
- `POST /notifications/{id}/read`
- `GET  /feed` ‚Äî activity feed from friends

### Standard error responses
- `401` ‚Äî unauthenticated / token invalid ‚Üí send to login (or refresh, if used)
- `403` ‚Äî forbidden (e.g. private profile)
- `404` ‚Äî not found
- `409` ‚Äî conflict (e.g. game already in library)
- `422` ‚Äî validation error ‚Üí `{ message, errors: { field: [..] } }`
- `429` ‚Äî rate limited ‚Üí `Retry-After` header

## Feature scope by phase (mobile)

Mobile follows the same roadmap as the web app but consumes the shared API. Suggested order:

1. **Auth** ‚Äî welcome, login, sign up, forgot/reset password, (optional) email verify, social login. *(UI already designed in Stitch.)*
2. **Personal library** ‚Äî add games, set status, filter list, edit entry details. **This is the core ‚Äî ship it first.**
3. **Discovery / Home** ‚Äî releases, upcoming, trending, search ‚Üí game detail ‚Üí add to library.
4. **Public profiles + stats + friends**.
5. **Community** (forum-lite).
6. **Messaging + realtime** (Reverb) + push notifications.
7. **Physical collection / setup** (image upload).

## Mobile MVP (first shippable slice)

The smallest version that proves the product on a device:

- Auth (login / register / token persistence)
- Search a game ‚Üí add to library with a status
- View "My Library" with status filter
- Edit a library entry (status, rating, review)
- Basic profile screen with sign out

Everything else (discovery feeds, friends, chat, community, collection) layers on after.
